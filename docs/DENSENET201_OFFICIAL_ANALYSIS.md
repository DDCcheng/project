# DenseNet-201 Member 2 Delivery

## Scope

This document records only the DenseNet-201 feature-extraction work. The
dataset audit, class mapping, and split are taken from the Part 1 pipeline and
were not modified by Member 2.

## Official data source

- Split source: `docs/file_list.csv`, generated by
  `src/preprocessing/make_split.py` on the project main branch.
- Task: binary `fresh/rotten` classification.
- Original source labels remain the 20 labels in `file_list.csv`; binary
  labels are derived only for the downstream binary experiments:
  `fresh* -> fresh`, `rotten* -> rotten`.
- The Part 1 image check reported 12,000 readable images, 0 corrupt images,
  and 20 original classes.
- The official split contains 8,399 training images, 1,801 validation images,
  and 1,800 test images.

| Split | Total | Fresh | Rotten | Ratio |
|---|---:|---:|---:|---:|
| Train | 8,399 | 4,276 | 4,123 | 70.0% |
| Validation | 1,801 | 918 | 883 | 15.0% |
| Test | 1,800 | 917 | 883 | 15.0% |

The file-list order is preserved exactly during feature extraction. Member 2
does not create another split, remove files, or change the Part 1 labels.

## Feature extractor

- Model: torchvision DenseNet-201 with `DenseNet201_Weights.DEFAULT`.
- The final classifier is removed and the global average-pooled feature is
  saved.
- Feature dimension: **1,920**.
- Parameters are frozen; extraction uses `model.eval()` and
  `torch.no_grad()`.
- Every image is converted to RGB, resized/transformed to 224x224, and
  normalized with ImageNet mean `(0.485, 0.456, 0.406)` and standard deviation
  `(0.229, 0.224, 0.225)`.

The implementation is
`src/feature_extraction/extract_densenet201_official.py`.

## Feature cases

### no_aug

All three splits use the base evaluation preprocessing.

### aug

Only the training split uses one deterministic, light augmentation view per
original image:

- `RandomResizedCrop(224, scale=(0.85, 1.0))`
- `RandomHorizontalFlip(p=0.5)`
- `RandomRotation(10 degrees)`
- light `ColorJitter`

The augmentation seed is derived deterministically from seed 42 and the row
index. Validation and test use the same non-augmented preprocessing as in the
`no_aug` case; their saved arrays are reused rather than regenerated.

## Outputs and extraction time

Each case is stored in its own directory under `results/features/` and
contains train, validation, and test feature arrays, original-label arrays,
binary-label arrays, and metadata.

| Case | Split | Rows | Feature dimension | Time |
|---|---|---:|---:|---:|
| no_aug | Train | 8,399 | 1,920 | 477.813 s |
| no_aug | Validation | 1,801 | 1,920 | 99.174 s |
| no_aug | Test | 1,800 | 1,920 | 85.902 s |
| aug | Train | 8,399 | 1,920 | 538.069 s |
| aug | Validation | 1,801 | 1,920 | reused from no_aug |
| aug | Test | 1,800 | 1,920 | reused from no_aug |

The extraction was run on CPU. The augmentation run took longer because the
training images were transformed before feature extraction.

## Validation performed

- Existing project feature validation passed for both `no_aug` and `aug`.
- Train, validation, and test row counts match `docs/file_list.csv`.
- All feature arrays have dimension 1,920.
- Binary labels match the official file-list order and contain only `fresh`
  and `rotten`.
- `no_aug` and `aug` training features are different, confirming that the
  training augmentation was applied.
- `no_aug` and `aug` validation arrays are identical.
- `no_aug` and `aug` test arrays are identical.
- No classification metrics are reported in this stage. Accuracy, precision,
  recall, F1-score, and confusion matrices are **待运行** in the PCA and
  classifier stage.
